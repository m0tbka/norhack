## Задачи приложения

Кейс подразумевает работу с большим массивом разнородной научно-технической литературы (доклады, журналы, материалы конференций, обзоры, статьи — суммарно около **2000 файлов** в форматах `.pdf`, `.doc`, `.xlsx`, `.pptx`, `.rar` и др.). Приложение решает следующие задачи:

| Задача | Описание |
|---|---|
| **Нормализация документов** | Автоматическое извлечение текста из разнородных офисных форматов и архивов, конвертация в единый машиночитаемый формат `.md` |
| **Построение графа знаний** | Извлечение сущностей (материалы, процессы, оборудование, свойства, эксперименты, публикации, эксперты, объекты инфраструктуры) и связей между ними из русскоязычных научных текстов по расширенной доменной схеме на базе NEREL |
| **Индексация и консолидация знаний** | Дедупликация и суммаризация сущностей/связей, построение сообществ в графе (Leiden-кластеризация) для получения локального и глобального контекста |
| **Интеллектуальный поиск и Q&A** | Ответы на вопросы пользователя по корпусу документов с опорой на граф знаний, локальный векторный поиск и веб-поиск как дополнительный источник |
| **On-premise развёртывание** | Весь стек (OCR → LLM-экстрактор → граф → поиск) может разворачиваться в контуре компании без передачи данных вовне |
| **Интерфейс исследователя** | Интерактивная визуализация графа знаний и поисковый UI без деградации производительности на графах с тысячами узлов |

## Архитектура системы

Решение состоит из двух связанных общим индексом контуров: **контур индексации** (offline, разовая/периодическая обработка корпуса) и **контур поиска** (online, обработка пользовательских запросов). Оба контура опираются на движок **RAGU** и модель **Meno-Lite-0.1** в качестве LLM-ядра.

<img width="1657" height="949" alt="Architecture" src="https://github.com/user-attachments/assets/1a5dc7e6-050d-44ff-99a5-1acfabd62bae" />

Оба контура напрямую соответствуют пайплайну, описанному в публикации RaguTeam (EMNLP-2026 System Demo): единый `Ingestion Pipeline` на этапе индексации и набор поисковых движков (`LocalSearch`, `GlobalSearch`, `NaiveSearch`, `MixSearch`, `QueryPlanEngine`) на этапе поиска.

### Детальная архитектура ingestion-пайплайна RAGU


<img width="1849" height="851" alt="Detailed" src="https://github.com/user-attachments/assets/88bca957-685f-4d0e-bdf8-e3ad53dd84c1" />

Ключевой архитектурный элемент — **двухэтапный экстрактор** (Stage 1 / Stage 2), который отличает решение от однопроходных GraphRAG-систем (например, LightRAG):

```
                      ┌────────────────────────────┐
  ЧАНК ТЕКСТА  ────▶  │  STAGE 1: Entity Extractor  │ ────▶  Провалидированный
                      │  (валидация по схеме NEREL+)│         список сущностей
                      └────────────────────────────┘                │
                                                                     ▼
                      ┌─────────────────────────────────────────────────────┐
                      │  STAGE 2: Relation Extractor                        │
                      │  связи извлекаются ТОЛЬКО между уже найденными      │
                      │  и провалидированными сущностями                    │
                      └─────────────────────────────────────────────────────┘
```

Такое разделение исключает несогласованные пары «сущность–связь» (когда связь ссылается на сущность, которой нет в графе) и повышает точность извлечения по сравнению с однопроходной экстракцией.

### Компонентная схема

| Слой | Компонент | Технология | Роль |
|---|---|---|---|
| Препроцессинг | Конвертация документов | **OCR (EasyOCR)** | Приведение `.pdf/.doc/.xlsx/.pptx/.rar/.zip` к единому `.md` |
| LLM-ядро | Извлечение сущностей/связей, генерация ответов | **Meno-Lite-0.1 (7B)** | Локальная модель-экстрактор и генератор, работает on-premise через vLLM |
| Оркестрация индексации и поиска | GraphRAG-движок | **RAGU** | Чанкинг, двухэтапная экстракция, консолидация, Leiden-кластеризация, поисковые движки |
| Схема разметки | Доменная онтология | **NEREL+** | Типы сущностей и связей, специфичные для металлургии/химии/материаловедения |
| Хранилище | Граф / KV / вектор | NetworkX–Neo4j / JSON–KV / NanoVDB–Qdrant (абстракция RAGU) | Персистентность графа знаний и индексов |
| Интерфейс | Визуализация и поиск | **$mol** | Ленивый рендеринг графа знаний, поисковая консоль |

## Алгоритм работы системы

Ниже описан пошаговый алгоритм — от загрузки «сырых» документов до ответа на вопрос пользователя.

### Этап A. Индексация корпуса (выполняется офлайн, при добавлении новых документов)

| Шаг | Операция | Инструмент | Результат |
|---|---|---|---|
| A.1 | Загрузка исходных файлов, распаковка `.zip`/`.rar` | `load_data.py` (OCR) | Список файлов на диске |
| A.2 | Оптическое распознавание и извлечение текста/таблиц из `.pdf/.doc/.xlsx/.pptx` | EasyOCR-пайплайн (`pipline.py`) | Единый набор `.md`-документов |
| A.3 | Чанкинг документов на фрагменты | `SimpleChunker` / `SemanticTextChunker` / `SmartSemanticChunker` (RAGU) | Список чанков с детерминированным ID `chunk-<MD5(текст)>` |
| A.4 | **Stage 1** — извлечение сущностей из каждого чанка и их валидация по схеме NEREL+ | Meno-Lite-0.1 + Artifact Extractor (RAGU) | Список типизированных сущностей на чанк |
| A.5 | **Stage 2** — извлечение связей, ограниченных множеством валидных сущностей из шага A.4 | Meno-Lite-0.1 + Relation Extractor (RAGU) | Список связей «сущность → сущность» с типом отношения |
| A.6 | Группировка одинаковых сущностей/связей из разных чанков (по имени и типу) | `EntitySummarizer` / `RelationSummarizer` | Единая карточка сущности/связи с объединёнными упоминаниями |
| A.7 | Для сущностей с большим числом дублей — кластеризация и LLM-суммаризация описаний | DBSCAN + LLM-суммаризатор | Консолидированное описание сущности |
| A.8 | Построение графа знаний, удаление изолированных узлов | Graph Constructor (RAGU) | Очищенный граф «Entity–Relation» |
| A.9 | Иерархическая кластеризация графа на сообщества | Leiden-алгоритм | Дерево сообществ (`community-<level>:<cluster_id>`) |
| A.10 | Генерация текстового описания каждого сообщества (заголовок, резюме, находки) | LLM-суммаризатор сообществ | Community reports |
| A.11 | Запись графа, KV-хранилища и векторного индекса | Storage-абстракция RAGU (Graph DB / KV / Vector Store) | Персистентный индекс, готовый к поиску |

### Этап B. Обработка пользовательского запроса (выполняется онлайн, в реальном времени)

| Шаг | Операция | Инструмент | Результат |
|---|---|---|---|
| B.1 | Приём вопроса пользователя через веб-интерфейс | $mol UI | Строка запроса |
| B.2 | (Опционально) декомпозиция сложного запроса на подзапросы | `QueryPlanEngine` (RAGU) | План выполнения запроса |
| B.3 | Поиск релевантных сущностей/сообществ/чанков в индексе | `LocalSearch` (векторный поиск по сущностям + связям + чанкам), `GlobalSearch` (по суммаризациям сообществ), `NaiveSearch` (плоский векторный RAG) или `MixSearch` (комбинация движков) | Набор релевантных фрагментов графа и текста |
| B.4 | (Опционально) дополнение контекста результатами веб-поиска | Web-модуль | Расширенный контекст |
| B.5 | Генерация ответа на основе собранного контекста | Meno-Lite-0.1 / LLM RAGU | Текстовый ответ пользователю |
| B.6 | Отображение ответа и, при необходимости, релевантного фрагмента графа знаний | $mol UI | Визуализация ответа + граф |

### Полный поток данных (по данным публикации RaguTeam, Figure 3)

```
Documents ──▶ List[str]
     │ Chunker.split
     ▼
List[Chunk]  (id = chunk-<MD5(content)>)
     │ Two-stage extraction
     ▼
EntitiesExtractionModel / RelationsExtractionModel
     │
     ▼
List[Entity]  (id = ent-<MD5(name+type)>)
List[Relation] (id = rel-<MD5(src+dst+type)>)
     │ Summarization / consolidation
     ▼
Consolidated List[Entity], List[Relation]
     │ Graph builder modules (например, RemoveIsolatedNodes)
     ▼
Filtered List[Entity], List[Relation]
     │ Hierarchical Leiden
     ▼
List[Community]  (id = com-<MD5(level:cluster_id)>)
     │ Community summarization
     ▼
List[CommunitySummary]
     │ Index persistence (upsert + merge policies)
     ▼
graph.gml + kv_*.json + vdb_*.json
```

Каждый доменный объект (сущность, связь, чанк, сообщество) получает детерминированный MD5-идентификатор, что позволяет выполнять **инкрементальные** upsert/update/delete без полной переиндексации при добавлении новых документов.

#### Локально через MAM dev-server

```bash
# в корне MAM-репо
git clone https://github.com/hyoo-ru/mam.git
cd mam
git clone https://github.com/b-on-g/norweb.git bog/norweb
npm install
npx mam start
```

Открыть: **http://localhost:9080/bog/norweb/front/app/-/test.html**

Dev-сервер на лету пересобирает при изменении любого `.view.tree` / `.view.ts` / `.view.css.ts` / `.locale=*.json`.

#### Сборка standalone

```bash
npx mam bog/norweb/front/app
```

Артефакт: `bog/norweb/front/app/-/` — статичная сборка для деплоя (index.html + web.js + web.css + локали).

#### Тесты

```bash
node bog/norweb/front/app/-/node.test.js
```

Должно вывести `All tests passed`. Текущий счёт — **283** (структурные snapshot'ы, e2e-flow через view-API, URL state, CSS rule intent, новые фичи).

Swagger UI: **http://localhost:8000/docs**. Все API-ручки находятся под `/api/v1`; upload/job/live-indexing endpoints отсутствуют, их доступность отражается через `GET /api/v1/capabilities`.

## RAGU — GraphRAG-движок

**RAGU** — открытый GraphRAG-движок с модульной многошаговой архитектурой построения графа знаний, лежащий в основе индексации и поиска в решении.

### Пайплайн индексации (6 конфигурируемых стадий)

| № | Стадия | Варианты реализации |
|---|---|---|
| 1 | Чанкинг | `SimpleChunker` (фиксированные окна), `SemanticTextChunker` (по семантическим границам), `SmartSemanticChunker` (с кросс-энкодерным реранкингом) |
| 2 | Извлечение сущностей и связей | Однопроходное (одним промптом) или **двухэтапное**: сначала сущности с валидацией, затем связи, ограниченные подтверждённым множеством сущностей |
| 3 | Суммаризация сущностей/связей | Группировка по (имя, тип); при большом числе дублей — DBSCAN-кластеризация + LLM-суммаризация |
| 4 | Построение графа | Удаление изолированных узлов, доп. модули очистки |
| 5 | Детекция сообществ | Иерархическая Leiden-кластеризация |
| 6 | Суммаризация сообществ | LLM генерирует структурированные отчёты (заголовок, резюме, находки) |

### Поисковые движки

| Движок | Принцип работы |
|---|---|
| `LocalSearch` | Векторный поиск по сущностям, расширенный на связанные связи и чанки |
| `GlobalSearch` | Поиск по LLM-суммаризациям сообществ (для широких, обзорных вопросов) |
| `NaiveSearch` | Классический плоский векторный RAG (baseline) |
| `MixSearch` | Параллельный запуск нескольких движков с объединением результатов |
| `QueryPlanEngine` | Декомпозиция сложного запроса на подзапросы (DAG) |

### Инженерные особенности

- Полностью открытый стек, разворачивается **on-premise** — данные не покидают контур.
- Трёхуровневая абстракция хранилища (`BaseStorage → graph/KV/vector`) с возможностью замены бэкенда (NetworkX → Neo4j, NanoVDB → Qdrant) без переписывания пайплайна.
- Асинхронный API с ограниченной конкурентностью, синхронные обёртки генерируются автоматически.
- Инкрементальные upsert/update/delete на основе детерминированных хэш-идентификаторов и политик слияния.
- Структурированный вывод LLM валидируется через Pydantic v2 — без ручного постпроцессинга JSON.

**Итог:** RAGU повышает Answer Correctness в **2–4×** относительно LightRAG (с ростом преимущества по мере усложнения задач — от ×2.1 на Fact Retrieval до ×4.1 на Creative Generation), а Evidence Recall — с 71–76% (LightRAG) до 76–84%. При этом размер LLM-экстрактора (3B → 14B) практически не влияет на итоговое качество графа (разброс Answer Correctness ≤1 п.п.), что подтверждает гипотезу «языковых навыков» и оправдывает использование компактной модели Meno-Lite-0.1.

## Meno-Lite-0.1 — компактная LLM для RAG-пайплайна

**Meno-Lite-0.1** — открытая 7B-модель, специально адаптированная для работы внутри RAG/GraphRAG-пайплайна: используется на этапе индексации для извлечения сущностей и связей, а также для генерации ответов тем же открытым стеком.

### Родословная и обучение

```
Qwen2.5-7B-Instruct → T-lite-it-1.0 → RuadaptQwen2.5-7B-Lite-Beta → Meno-Lite-0.1
```

| Стадия обучения | Объём данных | Состав |
|---|---|---|
| Продолженное претренирование | ~1.3 млрд токенов | Русско-английская смесь: FineWeb-Edu, отфильтрованный по сходству через `gte-multilingual-base` RuLM, российские образовательные PDF (FinePDFs-edu), тексты RuREBus |
| Контролируемое дообучение (SFT) | ~50 млн токенов | Инструкции по извлечению сущностей/связей на схеме NEREL, многошаговые QA-датасеты (MultiHop-RAG, MTRAGEval), логи запросов LightRAG по статьям Habr |

**Идея дизайна:** языковые навыки (извлечение информации, работа с контекстом, рассуждение над предоставленными фактами) насыщаются гораздо медленнее с ростом числа параметров, чем фактические «мировые» знания. Поэтому компактная модель, обученная работать с предоставленным контекстом, а не запоминать факты, конкурентоспособна внутри RAG-пайплайна по сравнению с моделями в разы большего размера.

### Метрики (по данным публикаций и карточки модели)

**Эффективность токенизатора:**

| Язык | Meno-Lite-0.1 | Базовый Qwen2.5 | Прирост |
|---|---|---|---|
| Русский | 3.77 симв./токен | 2.57 симв./токен | **+47%** |

## NEREL+ — расширенная схема разметки сущностей и связей

**NEREL** — открытый датасет с аннотациями вложенных именованных сущностей, связей и событий на русском языке.

| Показатель | Значение |
|---|---|
| Аннотированных именованных сущностей | 56 000 |
| Аннотированных связей | 39 000 |
| Типов сущностей | 29 |
| Типов отношений | 49 |

Для задач кейса схема расширена доменными сущностями и связями (**NEREL+**), задаваемыми на уровне промпта — без дообучения модели, что позволяет гибко расширять схему под новые задачи:

| Категория | Добавленные типы |
|---|---|
| Сущности | `MATERIAL`, `PROCESS`, `EQUIPMENT`, `PROPERTY`, `EXPERIMENT`, `PUBLICATION`, `EXPERT`, `FACILITY` |
| Связи | `USES_MATERIAL`, `OPERATES_AT_CONDITION`, `PRODUCES_OUTPUT`, `DESCRIBED_IN`, `VALIDATED_BY`, `CONTRADICTS` |

Оригинальная статья: *NEREL: A Russian dataset with nested named entities, relations and events* (RANLP 2021), arXiv:2108.13112.
Репозиторий датасета: https://github.com/nerel-ds/NEREL

## OCR — препроцессинг документов

Технология преобразования изображений текста (сканов, фотографий документов, PDF-страниц) в машиночитаемый текст на базе **EasyOCR**.

### Возможности

- Обработка файлов `.docx`, `.pdf`, `.xls`, `.pptx`, а также архивов `.rar`, `.zip` (с автоматическим извлечением содержимого).
- Приведение всех документов к единому формату `.md`.
- Запуск через скрипты `load_data.py` и `pipline.py`, с возможностью настройки под конкретные папки и вычислительные мощности.
- Корректное сохранение таблиц; извлечение картинок и поддержка `.xlsx`/`.pptx` находятся в разработке.

### Метрики производительности

| Показатель | Значение |
|---|---|
| Скорость обработки | ~1 МБ / 90 страниц за 1.5 минуты |
| Пример: PDF на 93 страницы (~7 МБ) | ~4 минуты |
| Сохранность таблиц | Да |
| Извлечение изображений, поддержка .xlsx/.pptx | В разработке |

### Структура обработанного корпуса (~2000 файлов)

| Тип документа | Количество |
|---|---|
| Доклады | 16 |
| Журналы | 394 |
| Материалы конференций | 1 509 |
| Обзоры | 104 |
| Статьи | 60 |

Отдельно выделены публикации из профильных журналов:

| Журнал | Количество публикаций |
|---|---|
| Copper 2010 | 23 |
| ALTA Ni-Co-Cu | 46 |
| New Caledonia Nickel 2013 | 37 |

### Поддерживаемые типы исходных файлов

`.pdf`, `.doc`, `.xlsx`, `.pptx`, `.rar`
